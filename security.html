Security

Browser and research-library access require explicit trust controls.

Notandia extensions and Zotero plugins execute in privileged environments. The project therefore treats permission scope, message validation, dependency risk, release provenance, update continuity, and incident reporting as product requirements.

Runtime principles

  • Request only permissions required for documented features.
  • Do not load remote executable code.
  • Validate messages, identifiers, URLs, colors, and bounded text before use.
  • Keep external requests bounded, time-limited, and user-controllable.
  • Do not write untrusted data into HTML execution sinks.
  • Separate platform packaging from shared detection and evidence logic.
  • Preserve released extension identities and persistent settings unless a tested migration exists.

Supply-chain and release controls

  • GitHub Actions are pinned to immutable commit SHAs.
  • Dependency audits and static analysis run in CI where applicable.
  • Packages are validated as archives before publication.
  • Checksums and provenance attestations are generated for supported releases.
  • Release workflows refuse version mismatches, prerelease store publication, and accidental overwrites.
  • Store publication uses separate protected environments and existing update identities.

Website security

The site is dependency-free static HTML and CSS. It loads no remote JavaScript. Cloudflare response headers enforce a restrictive content security policy, transport security, framing restrictions, MIME sniffing prevention, and a minimal permissions policy.

Reporting vulnerabilities

For sensitive security reports, use GitHub private vulnerability reporting in the affected repository. Include the affected version, platform, impact, reproduction steps, and a minimal proof of concept. Do not publicly disclose exploitable details before maintainers have had a reasonable opportunity to respond.
